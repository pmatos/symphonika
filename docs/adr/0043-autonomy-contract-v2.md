# Autonomy contract v2: ban GitHub MCP connector and self-applied `needs-human`

Symphonika bumps the autonomy preamble to `autonomy-preamble-v2` and extends `WORKFLOW.md` with three additional contract points after observing that the v1 contract under-specified what "autonomous" means in practice. Run `568cdd79` on issue #45 emitted "human help" and called the GitHub MCP connector tool `add_issue_labels`; the connector emitted `mcpServer/elicitation/request`, which `src/providers/codex.ts:491-497` classifies as `input_required`, ending the run with `terminal_reason="provider requested input"` and `sym:failed`. The `--dangerously-bypass-approvals-and-sandbox` flag only suppresses Codex's sandbox/exec approvals; it does not suppress per-tool elicitations from MCP connectors. The v2 contract therefore tells the agent (a) it is unattended and no operator will answer prompts, approve tool calls, or read intermediate output during the run; (b) every GitHub mutation must go through the local `gh` CLI (`gh issue ...`, `gh pr ...`, `gh issue comment ...`, `gh issue edit ...`), which has no elicitation surface — never the GitHub MCP connector tools (for example `add_issue_labels`, `create_pull_request`); (c) the agent must not self-apply `needs-human` as an exit strategy: if work cannot proceed it must leave a `gh issue comment` describing what blocked it and exit cleanly, leaving label triage to the operator. ADR 0017's v1 invariants (autonomous, full-permission, no operator input, preserve evidence, prepared workspace, assigned issue) remain valid and are restated verbatim in v2; this ADR extends them rather than supersedes them. ADR 0018 said Symphonika does not expose provider GitHub tools to the agent; this ADR addresses an agent-side gap 0018 did not anticipate — workflow contracts must also instruct the agent not to use any agent-environment-supplied GitHub MCP connector tools, because they elicit operator approval through the provider transport that 0018 specifically wanted to avoid. The preamble version field in `prompt-metadata.json` (see `src/workflow.ts:274-287`) becomes the evidence-replay signal for distinguishing v1 from v2 runs. `needs-human` remains a valid `labels_none` exclusion in service config (`SPEC.md:198, 417`); only the agent-side self-application is prohibited.
