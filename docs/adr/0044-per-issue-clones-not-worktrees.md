# Per-issue clones, not worktrees

Symphonika prepares each issue workspace as a per-issue `git clone --shared` of the bare repository cache rather than as a `git worktree` of it. The clone has a real `.git/` directory inside the workspace path, so the index, `index.lock`, and refs all live under the agent's cwd; sandboxes that scope writes to the workspace cwd (codex's default `workspace-write` mode and any equivalent locked-down provider) can therefore run `git add`, `git commit`, and `git push -u origin <branch>` without writing outside the workspace. The clone uses `--shared` so the object store is still cheaply shared with the cache via `objects/info/alternates`, and `origin` is repointed at the project's upstream remote immediately after the clone so the branch push reaches GitHub without further config. The bare cache, deterministic issue branch, and deterministic workspace path from ADR-0007 are unchanged; this ADR supersedes only the worktree mechanism in that decision. Two consequences are explicitly accepted: (1) the alternates link couples a workspace's object reachability to the cache's lifetime, so future cleanup tooling must gate cache deletion on "no remaining clones"; (2) `refs/remotes/origin/*` inside a fresh clone reflects the cache snapshot until the first `git fetch origin` against the upstream, which is acceptable because workflow contracts already drive their own fetch and PR logic. Historical ADRs 0030, 0040, and 0041 still use the word "worktree" in their recorded decision text; their intent carries over and "worktree" in those texts should be read as "the workspace mechanism then in use".
