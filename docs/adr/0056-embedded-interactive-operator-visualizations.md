# Allow embedded interactive visualizations in server-rendered operator pages

Symphonika permits a server-rendered operator page to embed a self-contained, client-side interactive visualization when it makes stored run evidence materially easier to explore, and narrows the SPEC §2 "rich frontend application" non-goal to mean a *separate/standalone* single-page application rather than any client-side interactivity. The motivating case is the workflow-graph view at `GET /runs/:id/graph` (PR #239, issue #242), which renders a run's persisted expanded FSM (ADR-0045) as a labeled state diagram with pan/zoom and click-to-inspect — for a workflow with back-edges and predicate-labeled transitions, a static list is genuinely harder to read than an interactive graph. The non-goal was written to keep v1 from growing a separate frontend application with its own build, routing, and state management (ADR-0010); embedding a visualization library in one server-rendered page does not do that, so treating "loads client-side JavaScript" as equivalent to "ships a rich frontend application" was too strict.

To keep this from eroding the server-rendered UI posture, an embedded visualization must satisfy four guardrails. First, the page stays server-rendered: the server produces the HTML and inlines the data it needs, rather than the client fetching and assembling the view. Second, it is self-contained — no build step and no bundled single-page application; scripts are either vendored or pinned CDN assets with Subresource Integrity, and the page is one route, not an app shell. Third, it degrades gracefully when its external visualization dependencies are unavailable: if the CDN or vendored viz libraries are blocked or fail Subresource Integrity, the page's own inline script renders a text listing of the underlying evidence (for the graph view, the FSM's states and transitions) instead of a blank canvas. (This guarantee covers a failed viz-library load, not fully disabled JavaScript; the inline script still runs and produces the fallback.) Fourth, it stays read-only — the visualization adds exploration affordances (pan, zoom, drag, click-to-inspect) but no mutating actions beyond the explicit active-run cancellation and manual poll-now trigger already allowed by ADR-0027; label creation, stale-claim reset, and workspace cleanup remain CLI-only.

This amends ADR-0010 and refines the operator-surface intent of ADR-0009 and the read-only stance of ADR-0027 without changing them: the CLI remains the primary and only mutation-complete surface, and the web UI remains an observability console that now includes an interactive graph view for the evidence persisted under ADR-0045. Anything beyond a self-contained, read-only, gracefully-degrading visualization — a standalone SPA, client-side routing, a persistent client application, or new mutating actions in the browser — remains out of scope and would need its own ADR.
